# AI Beauty Stylist 多语言上线验收计划

更新时间：2026-06-14

## 1. 验收目标

本计划用于在正式上线前，从产品经理、SEO、妆容专业顾问和营销增长四个角度，对 AI Beauty Stylist 项目进行整站验收。

验收重点不是单纯确认页面存在，而是确认所有语言版本都达到可上线标准：

- 用户能理解产品价值，并顺利完成核心使用路径。
- Google 能发现、抓取、理解并索引正确的多语言页面。
- 妆容建议内容专业、可信、不过度承诺。
- 页面具备清晰转化路径、信任信号和本地化表达。

## 2. 验收范围

以项目实际路由、构建产物、sitemap 和多语言配置为最终准入范围。

需要覆盖：

- 首页
- AI 妆容工具页
- 结果页或推荐页
- 场景/风格/关键词落地页
- FAQ 页面
- About / Contact / Privacy / Terms / Disclaimer 等信任页面
- 所有语言版本页面
- sitemap、robots、canonical、hreflang、结构化数据
- 移动端和桌面端核心体验
- 上线后的 Google Search Console 监控项

## 3. 验收原则

- 优先检查最终构建后的页面，而不是只看源码。
- 所有 indexable 页面必须有明确用户意图和搜索意图。
- 多语言页面不能只是机器直译，需要符合本地用户表达和搜索习惯。
- SEO 不能靠关键词堆砌，必须服务真实用户需求。
- 妆容建议不能制造外貌焦虑，不能做绝对化、医学化或无法证明的承诺。
- P0 和 P1 问题必须在上线前修复。

## 4. 四个验收视角

### 4.1 产品经理视角

核心问题：用户能不能快速理解并完成任务？

检查项：

- 首屏是否清楚说明产品能解决什么问题。
- 核心 CTA 是否明显，例如开始测试、上传照片、生成妆容建议。
- 核心路径是否顺畅：进入页面、理解功能、提交输入、查看结果、继续操作。
- 移动端是否可用，按钮、表单、图片和文字没有遮挡或错位。
- 异常状态是否完整，例如上传失败、生成失败、网络失败、空结果。
- 页面之间是否有清晰导航，不让用户进入死胡同。
- 隐私、免责声明和联系入口是否容易找到。

通过标准：

- 新用户无需额外说明即可理解产品。
- 核心功能路径无阻断。
- 所有主语言和非主语言版本都没有明显半成品感。

### 4.2 SEO 视角

核心问题：搜索引擎能不能正确抓取、理解和收录？

检查项：

- 每个语言版本是否有独立、稳定、可访问的 URL。
- 每个页面是否有唯一的 title、meta description 和 H1。
- canonical 是否指向当前页面的规范 URL。
- hreflang 是否完整、互相回链，并包含合理的 x-default。
- sitemap 是否只包含应该被索引的页面。
- robots.txt 是否没有误挡关键页面和资源。
- 页面是否存在 noindex 与 sitemap 冲突。
- 结构化数据是否 JSON-LD 可解析，且内容与页面一致。
- 内部链接是否能让重点页面被发现，而不是只靠 sitemap。
- 多语言页面是否有足够差异和本地化信息增益。

通过标准：

- sitemap 中所有 URL 都返回正常状态。
- indexable 页面没有 canonical、noindex、robots、hreflang 冲突。
- 重点页面都有明确搜索意图、唯一元信息和可见正文内容。

### 4.3 妆容大师视角

核心问题：内容是否像专业美妆顾问给出的建议？

检查项：

- 是否区分肤色深浅、冷暖调、肤质、脸型和使用场景。
- 是否避免“一套妆容适合所有人”的泛化建议。
- 是否能解释推荐依据，而不只是给出结果。
- 是否覆盖日常、职场、约会、派对、拍照、成熟肌、敏感肌等常见场景。
- 是否尊重不同肤色、年龄、性别表达和审美偏好。
- 是否避免外貌焦虑、羞辱性表达和绝对化承诺。
- 多语言美妆术语是否准确，例如 undertone、foundation shade、contour、blush、dewy、matte 等。

通过标准：

- 妆容建议有依据、有场景、有边界。
- 内容专业可信，不像关键词拼接或模板复制。
- 不出现可能损害用户信任或安全感的表达。

### 4.4 营销大师视角

核心问题：用户为什么愿意继续用、分享、注册或付费？

检查项：

- 首屏是否在 3 秒内说明产品价值。
- CTA 是否明确，并在关键位置自然重复。
- 是否展示示例结果、使用场景或前后对比。
- 是否有隐私和安全信任信号，例如照片处理说明、AI 建议边界。
- 是否有低门槛试用入口。
- 是否有保存结果、重新生成、分享结果或继续探索的动作。
- 不同语言版本的卖点是否符合当地市场表达。
- 页面文案是否有行动导向，而不是只陈述功能。

通过标准：

- 用户知道产品有什么用、怎么开始、下一步做什么。
- 页面具备信任感和转化动线。
- 非英文页面的营销表达自然，不是英文直译。

## 5. 执行流程

### 阶段 1：项目现状冻结

目标：确认验收基线，避免边改边验导致结果失真。

动作：

- 记录当前分支、提交状态和构建命令。
- 确认部署目标和正式域名。
- 确认所有计划上线的语言列表。
- 确认主语言、默认语言和 x-default 策略。
- 确认哪些页面应该收录，哪些页面应该 noindex 或不进 sitemap。

输出：

- 验收基线说明
- 语言列表
- 页面类型列表
- indexable / noindex 初始规则

### 阶段 2：全站 URL 和语言盘点

目标：拿到真实可上线页面清单。

动作：

- 从路由、多语言配置和 sitemap 中提取所有 URL。
- 按页面类型分类：工具页、落地页、内容页、法律页、系统页。
- 对比源代码路由和构建产物，找出缺失、重复或孤立页面。
- 标记每个 URL 的语言、canonical、hreflang、是否应收录。

输出：

| 页面类型 | 语言 | URL | 是否应收录 | canonical | hreflang | 状态 |
| --- | --- | --- | --- | --- | --- | --- |
| 首页 | 待填 | 待填 | 是 | 待验 | 待验 | 待验 |
| 工具页 | 待填 | 待填 | 是 | 待验 | 待验 | 待验 |
| SEO 落地页 | 待填 | 待填 | 待定 | 待验 | 待验 | 待验 |
| 法律页 | 待填 | 待填 | 待定 | 待验 | 待验 | 待验 |

### 阶段 3：技术 SEO 验收

目标：确认搜索引擎可以正确处理站点。

动作：

- 构建项目并检查最终 HTML。
- 检查 sitemap 是否完整、干净、无无效 URL。
- 检查 robots.txt。
- 批量检查 title、description、H1。
- 批量检查 canonical 和 hreflang。
- 检查 JSON-LD 是否可解析。
- 检查内部链接、图片链接和关键静态资源是否有效。
- 检查是否存在重复标题、重复描述、薄内容或模板高度重复页面。

建议命令：

```powershell
npm.cmd run build
Get-ChildItem -Recurse -File .\dist -Filter index.html
Get-Content -Raw .\dist\robots.txt
Select-String -Path .\dist\**\index.html -Pattern '<link rel="canonical"|hreflang|application/ld\+json|<h1'
```

输出：

- 技术 SEO 问题清单
- sitemap 准入清单
- canonical / hreflang 修复清单
- 结构化数据修复清单

### 阶段 4：多语言本地化验收

目标：确认所有语言版本都像真实面向当地用户发布的产品。

动作：

- 全量检查导航、按钮、表单、错误提示、核心功能文案。
- 抽样检查每种语言的首页、工具页、结果页、FAQ 和 SEO 落地页。
- 检查是否混入英文、简体中文、占位符或变量未替换。
- 检查美妆术语是否准确。
- 检查 title、description、H1 是否符合当地搜索习惯。
- 检查 CTA 是否自然，不机械翻译。

最低抽样标准：

- 首页：所有语言 100% 检查。
- 核心工具路径：所有语言 100% 检查。
- 法律和信任页面：所有语言 100% 检查。
- SEO 落地页：每种语言至少抽样 30%，重点关键词页 100% 检查。
- FAQ：每种语言至少抽样 50%。

输出：

- 多语言质量问题清单
- 术语表问题清单
- 需人工润色页面清单

### 阶段 5：妆容专业内容验收

目标：确认内容专业、可信、符合美妆用户期待。

动作：

- 检查妆容推荐逻辑是否有依据。
- 检查内容是否考虑肤色、冷暖调、肤质、场景和风格。
- 检查是否存在过度承诺或不安全建议。
- 检查图片、alt 和页面文案是否匹配。
- 检查结果页是否给出可执行建议，而不只是泛泛描述。

输出：

- 专业可信度问题清单
- 需补充的场景和肤色覆盖清单
- 高风险表达修复清单

### 阶段 6：转化和营销验收

目标：确认用户有理由开始使用，并知道下一步。

动作：

- 检查首页首屏价值表达。
- 检查每个重点页面的 CTA。
- 检查示例结果和信任信号。
- 检查是否有保存、分享、重新生成、继续探索等留存动作。
- 检查不同语言的卖点是否本地化。
- 检查 Open Graph 和 Twitter Card 展示效果。

输出：

- 转化路径问题清单
- 首屏和 CTA 优化建议
- 社交分享展示问题清单

### 阶段 7：功能、性能和移动端验收

目标：确认真实用户访问不会遇到明显体验问题。

动作：

- 桌面端和移动端打开重点页面。
- 验证核心功能路径。
- 检查页面加载、图片尺寸、懒加载和视觉稳定性。
- 检查按钮可点击区域、表单输入、错误提示。
- 检查 404 页面和异常状态。
- 检查基本可访问性：语义标题、alt、颜色对比、键盘可用性。

输出：

- 端到端功能验收记录
- 移动端问题清单
- 性能和可访问性问题清单

### 阶段 8：上线后监控

目标：上线后持续确认真实搜索和用户表现。

动作：

- 提交 sitemap 到 Google Search Console。
- 检查重点页面 URL Inspection。
- 观察索引覆盖、hreflang、抓取异常和重复页面。
- 监控品牌词、核心关键词、长尾关键词表现。
- 检查 GA 或其他分析工具的核心事件。
- 记录上线后 7 天、14 天、30 天的 SEO 和转化数据。

输出：

- 上线后监控表
- GSC 问题清单
- 第一轮增长优化清单

## 6. 优先级定义

### P0：上线阻断

必须上线前修复。

- 核心页面无法访问或构建失败。
- robots、noindex、canonical 阻止重点页面收录。
- hreflang 大面积错误。
- sitemap 包含大量无效 URL。
- 核心功能无法完成。
- 存在严重误导、安全风险或损害信任的内容。

### P1：上线前必须修复

不一定导致全站不可用，但会明显影响收录、转化或信任。

- 重点页面 title、description、H1 缺失或重复。
- 多语言页面明显机器翻译或混入错误语言。
- 重点 CTA 不清晰。
- 妆容建议过于泛化或专业性不足。
- 重点页面正文薄、重复度高、没有信息增益。
- 移动端核心路径存在明显体验问题。

### P2：建议上线前修复，可排期

- 局部文案不够自然。
- 部分图片 alt 不完整。
- 非核心页面内链不足。
- 结构化数据可增强。
- 性能有优化空间，但不阻断核心体验。

### P3：上线后增长优化

- 新增内容集群。
- 扩展长尾关键词页面。
- 增加案例、示例妆容和对比内容。
- 优化转化实验。
- 补充更多本地化营销素材。

## 7. 上线准入标准

满足以下条件后，才建议正式上线：

- P0 问题数量为 0。
- P1 问题数量为 0，或有明确原因、负责人和上线后修复时间。
- 构建成功，重点页面可访问。
- sitemap、robots、canonical、hreflang 无阻断性错误。
- 所有语言首页和核心路径通过人工验收。
- 重点 SEO 页面有唯一标题、描述、H1 和可见正文。
- 核心妆容建议内容专业、可信、无高风险表达。
- 移动端核心路径可完成。
- 隐私、免责声明和联系入口完整。
- 上线后 GSC 和数据监控方案已准备好。

## 8. 最终交付物

上线验收应产出以下文档或记录：

- 全站 URL 和语言版本清单
- SEO 技术验收报告
- 多语言本地化验收报告
- 妆容专业内容问题清单
- 营销转化优化清单
- P0/P1/P2/P3 问题优先级表
- 上线准入结论
- 上线后 7/14/30 天监控计划

## 9. 下一步执行建议

建议按以下顺序推进：

1. 先跑构建并生成真实页面清单。
2. 从 sitemap 和构建产物反推所有可上线 URL。
3. 批量检查 SEO 技术项，先清掉 P0/P1。
4. 对所有语言首页和核心路径做人工验收。
5. 对 SEO 落地页做抽样和重点页全量检查。
6. 汇总问题优先级，修复上线阻断项。
7. 复测构建产物和浏览器真实页面。
8. 上线后接入 GSC 监控，不把 sitemap 提交当作收录保证。
