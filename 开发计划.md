## 不可违反的开发约束

1. 根目录以下 8 个 HTML 页面是产品开发基线，必须在它们现有的信息架构、页面布局、视觉方向和用户状态基础上开发，禁止丢弃后从空白模板重做：
   - `index.html`
   - `discover.html`
   - `diagnosis.html`
   - `pricing.html`
   - `share-card.html`
   - `tryon-free.html`
   - `tryon-pro.html`
   - `tryon-premium.html`
2. `UI/` 目录的 8 张页面截图是视觉验收基准。功能开发允许补充状态和组件，但不得自行改成另一套设计风格。
3. 迁移到 Astro 时必须逐页、逐区块迁移上述 HTML；根目录原型在迁移验收完成前保留，不能先删除再重建。
4. 首发优先完成这 8 个核心页面的真实闭环。法律页、支付回调、账户中心等支持页面或 API 路由可以新增，但不能用新增页面替代这 8 个核心页面。
5. 共享导航、按钮、卡片、色彩、圆角和会员状态必须复用统一设计令牌与组件，不能让 8 个页面继续各自演化。
6. 当前简体中文页面既是结构和视觉基线，也是第一阶段正式开发语言；首发根站默认使用简体中文，同时预埋 i18n，中文闭环完成前不开发其他语言。
7. Free 额度以完整 PRD 为准：每月基础 3 次，分享每天最多奖励 1 次；生成失败不扣额度或自动返还。
8. 后续页面、妆容库和推荐模块优先直接使用 `images/` 内现有图片。除非现有素材没有可用候选且用户明确确认，否则不得重新生成妆容图。
9. `images/` 文件名是妆容或风格的稳定资产标识；建立 catalog 时直接按文件名映射 slug。允许按展示需要生成尺寸衍生图，但不得擅自替换原图、改名或删除。

## 当前基线审计

| 页面 | 当前已有基础 | 开发重点 |
|---|---|---|
| `index.html` | Hero、前后对比视觉、精选妆容入口 | 接入统一妆容库、真实参数跳转、补齐场景/流程/信任/会员价值区 |
| `discover.html` | 筛选器、妆容列表、试妆入口 | 真实筛选、统一 catalog、保留 `look`/`scenario` 参数、埋点 |

| `diagnosis.html` | 诊断报告静态样例、推荐妆容 | 接入真实结构化诊断、置信度、免责声明、推荐逻辑、新诊断流程 |
| `tryon-pro.html` | Pro 结果工作台、产品类型、分享入口 | 权限校验、真实结果、保存/下载、历史、完整教程和预算工具包 |
| `tryon-premium.html` | Premium 工作台、增强权益、分享入口 | 优先队列、长期历史、完整妆容库、风格档案和动态建议 |
| `share-card.html` | 品牌分享卡静态版 | 动态数据、图片导出、Web Share、回流链接、奖励幂等 |
| `pricing.html` | 三档价格、功能对比、FAQ | Stripe Checkout/Portal、当前计划状态、年付、退款/续费告知、结构化数据 |

当前技术基线：

- 页面使用静态 HTML + `styles.css` + `app.js`。
- 现有交互大多只显示 Toast，尚未接入真实业务状态。
- 根目录暂无 Astro、测试、构建、数据库迁移或 Cloudflare 配置。
- 页面本地资源链接完整，`images/` 已具备 50 张可直接使用的有效素材。
- 当前无 Git 仓库，实施前需要建立版本管理。

| 项目 | 结果 |

| 可直接使用 | 50 张：3 张 Hero、44 张妆容图、3 张风格图 |

| 尚未被 8 个页面使用 | 43 张有效素材，可直接用于扩充首页、发现页和试妆页 |
| 常见尺寸 | 720×1280、1024×1024、864×1821 等，均可通过响应式容器和 `object-fit` 直接使用 |

图片使用决定：

- 首发妆容库直接从这 50 张有效素材中组织，不安排重新生成妆容图片。
- 页面需要新妆容卡时，优先按 `look-*` 文件名建立妆容名称、分类、场景和说明。
- `hero-*` 用于首页主视觉与信任展示，`style-*` 用于风格入口，`look-*` 用于妆容 catalog。
- 只有现有图片无法满足明确的新场景，且用户确认后，才新增图片生成任务。
